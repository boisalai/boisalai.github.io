Projet

Objectif du projet​

Le projet vise à développer un modèle prédictif pour les interactions futures des utilisateurs sur un site de commerce électronique, utilisant un large ensemble de données fourni par Coveo. Cette base de données comprend plus de 30 millions d'interactions avec des produits, enrichies de métadonnées cataloguées et de comportements de recherche détaillés. Elle inclut non seulement les clics sur les produits après une recherche, mais aussi les visualisations sans clics, offrant une perspective unique sur la rétroaction négative des utilisateurs. L'enjeu principal est de prédire les interactions des utilisateurs sur la base de leurs actions précédentes et de leurs recherches au sein d'une session.

Données​

Notre projet utilise les données issues de la compétition SIGIR eCOM 2021, organisée par Coveo. Cette base de données, disponible pour des fins de recherche, a été publiée dans le cadre du Data Challenge SIGIR eCom 2021 et est accessible publiquement pour des utilisations académiques et de recherche. Notre utilisation de ces données suit les termes et conditions établis par Coveo, comme spécifié dans leur déclaration de licence. Ces données jouent un rôle essentiel dans notre projet, nous permettant de développer et de tester nos solutions avec un ensemble de données riche et varié, représentatif des défis réels du domaine du commerce électronique.

Ces données sont constituées de trois fichiers CSV, chacun offrant des perspectives uniques et complémentaires sur le comportement des utilisateurs et les caractéristiques des produits dans le cadre du commerce électronique :

  • browsing_train.csv : Ce fichier contient des données sur le parcours de navigation des utilisateurs sur le site, incluant des informations telles que les clics, les ajouts au panier, et les achats. Il fournit une vue détaillée sur le comportement des utilisateurs en interaction avec les produits.
  • search_train.csv : Ce fichier se concentre sur les activités de recherche des utilisateurs, enregistrant les requêtes saisies et les interactions avec les résultats de recherche. Il est essentiel pour comprendre comment les utilisateurs cherchent des produits et réagissent aux résultats présentés.
  • sku_to_content.csv : Ce fichier fait le lien entre les identifiants des produits (SKU) et leur contenu descriptif. Il offre une compréhension détaillée des caractéristiques des produits, essentielle pour analyser la relation entre les attributs des produits et le comportement des utilisateurs.

Plus concrètement, le fichier browsing_train.csv compte 36 079 307 événements de navigation dans une session sur un site de commerce électronique. Les colonnes de ce fichier sont :

  • session_id_hash (string) : Identifiant haché de la session d'achat.
  • event_type (enum) : Type d'événement. Soit pageview pour la consultation d'une page ou event_product pour une action sur un produit.
  • product_action (enum) : Si event-type est égal à event_product, alors cette colonne contient l'action sur un produit. Soit detail pour consulter la fiche descriptive d'un produit, add pour ajouter un produit dans un panier d'achat, purchase pour acheter un produit, remove pour supprimer un produit du panier d'achat.
  • product_sku_hash (string) : Si product_action non vide, alors cette colonne contient l'identifiant haché du produit de l'événement.
  • server_timestamp_epoch_ms (int) : Horodatage en millisecondes.
  • hashed_url (string) : URL hachée de la page web.

Le fichier search_train.csv contient 819 516 interactions basées sur la recherche. Chaque ligne est un événement de requête de recherche émis par un utilisateur, qui comprend un tableau de résultats. Les colonnes de ce fichier sont :

  • session_id_hash (string) : Identifiant haché de la session d'achat.
  • server_timestamp_epoch_ms (int) : Horodatage en millisecondes.
  • query_vector (vector) : Une représentation dense de la requête de recherche, obtenue grâce à des techniques standard de modélisation pré-entraînée et de réduction de dimensionnalité.
  • product_skus_hash (list) : Identifiants hachés des produits affichés dans la réponse de recherche.
  • clicked_skus_hash (list) : Identifiants hachés des produits cliqués après avoir émis la requête de recherche.

Le fichier sku_to_content.csv contient des informations sur 66 386 produits dont des représentations dense des métadonnées textuelles et images du catalogue réel. Les colonnes de ce fichier sont :

  • product_sku_hash (string) : Identifiant haché du produit.
  • category_hash (string) : Les catégories sont des représentations hachées de la hiérarchie des catégories, séparées par le caractère '/'.
  • price_bucket (int) : Le prix du produit, fournir sous forme d'entier de 10 quantiles.
  • description_vector (vector) : Une représentation dense de métadonnées textuelles, obtenue grâce à des techniques standard de modélisation pré-entraînée et de réduction de dimensionnalité, compatible avec celle du fichier de recherche.
  • image_vector (vector) : Une représentation dense des métadonnées d'image, obtenue grâce à des techniques standard de modélisation pré-entraînée et de réduction de dimensionnalité.

Démarche utilisée​

Ensuite, pour développer notre solution, nous avons réalisé les étapes suivantes :

  • Étape 0 - Initialisation : Importation des librairies requises, définition des répertoires, déclaration de fonctions utilitaires et création de sous-ensembles de dimensions gérables.
  • Étape 1 - Chargement des données : Chargement des fichiers browsing_train.csv, search_train.csv, sku_to_content.csv et création des DataFrames browsing, search et product.
  • Étape 2 - Analyse exploratoire des données (EDA) : Exploration des DataFrames, notamment leurs distributions, les valeurs manquantes, les anomalies, etc.
  • Étape 3 - Nettoyage des données : Gestion des valeurs manquantes, supression des doublons.
  • Étape 4 - Ingénierie des attributs : Convertion de chaînes de caractères en listes, création de caractéristiques.
  • Étape 5 - Fusion des données : Fusion des trois ensembles (navigation, recherche et produits) en un seul ensemble, où chaque ligne représente un événement d'une session.
  • Étape 6 - Agrégation après fusion : Agrégation des données par session, où chaque ligne contient toutes les séquences d'événements d'une session. D'autres caractéristiques sont également créées.
  • Étape 7 - Création des séquences et tokenisation : Création des séquences d'interaction pour l'entraînement, les interaction suivantes à prédire et les interactions futures pour l'évalation de la performance du modèle. Tokenisation de ces ensembles.
  • Étape 8 - Division en ensembles d'entraînement, de validation et de test
  • Étape 9 - Création du modèle : Création d'un modèle XLNet.
  • Étape 10 - Création des DataLoaders
  • Étape 11 - Entraînement du modèle : Entraînement du modèle sur l'ensemble d'entraînement et évaluation des performances sur l'ensemble de validation.
  • Étape 12 - Hyperparamétrage : Ajustement pour optimiser les performances.
  • Étape 13 - Évaluation sur l'ensemble de test : Mesurer l'efficacité sur des données non vues.
  • Étape 14 - Analyse d'erreur post-entraînement : Comprendre et améliorer les prédictions.

Chacune des étapes est décrite plus en détail ci-dessous. Après la plupart des étapes, nous récapitulons les transformations, sauvegardons et récupérons les données pour éviter d'avoir à recommencer depuis le début advenant un problème.

Étape 3 - Nettoyage des données​

Cette étapes vise à traiter les valeurs manquantes, les erreurs, les doublons, etc. Les ensembles transformés sont sauvegardés dans le répertoire de sortie pour usage futur.

L'exploration des données nous a révélé qu'une proportion élevée des interactions sont des événements de consultation de pages non liées à un produit. Pour ces événements, nous utilisons la page consultée comme un produit virtuel. Concrètement, nous créons une nouvelle caractéristique product_url_hash correspondant à l'identifiant du produit (product_sku_hash) s'il existe, sinon à la page consultée (hashed_url) dont le modèle aura pour tâche de prédire.

def clean_browsing(df: pd.DataFrame) -> pd.DataFrame:
    # Supprimer les lignes dont l'identifiant est manquant.
    condition = df["session_id_hash"].isnull()
    df = df.loc[~condition]

    # Supprimer les doublons en se basant sur plusieurs colonnes, en gardant le dernier.
    df.drop_duplicates(
        ["session_id_hash", "server_timestamp_epoch_ms", "event_type"],
        inplace=True,
        keep="last",
    )

    # Remplacer les valeurs manquantes dans 'product_action' par 'view'.
    df["product_action"].fillna("view", inplace=True)

    # Remplir les valeurs manquantes.
    df["hashed_url"].fillna("none", inplace=True)

    # Si l'identifiant du produit est manquant, lui attribuer 'hashed_url'.
    # Créer 'product_url_hash' correspondant donc à un produit ou à un URL.
    df["product_url_hash"] = df["product_sku_hash"].fillna(df["hashed_url"])

    return df

def clean_search(df: pd.DataFrame) -> pd.DataFrame:
    # Supprimer les lignes dont l'identifiant est manquant.
    condition = df["session_id_hash"].isnull()
    df = df.loc[~condition]

    return df

def clean_product(df: pd.DataFrame) -> pd.DataFrame:
    # Supprimer les lignes dont l'identifiant est manquant.
    condition = df["product_sku_hash"].isnull()
    df = df.loc[~condition]

    # Supprimer les doublons.
    df.drop_duplicates(["product_sku_hash"], inplace=True, keep="last")

    return df

Étape 4 - Ingénierie des attributs​

L'ingénierie des attributs, ou feature engineering en anglais, a pour but de créer de nouvelles variables (features) à partir de données brutes pour aider à améliorer la performance des algorithmes de prédiction. Ci-dessous, nous avons des fonctions utilitaires et des fonctions qui transforment les ensembles de données de navigation, de recherche et sur les produits.

import json
import pandas as pd

def convert_str_to_list(x):
    """Convertir une chaîne de caractères en liste."""
    if pd.isnull(x):
        return x
    return json.loads(x.replace("'", '"'))

def process_browsing(df: pd.DataFrame) -> pd.DataFrame:
    # Pour l'instant, aucune transformation n'est appliquée.
    # Le DataFrame est inchangé.
    return df

def process_search(df: pd.DataFrame) -> pd.DataFrame:
    # Convertir les chaînes de caractères en listes.
    for col in ["product_skus_hash", "clicked_skus_hash", "query_vector"]:
        df[col] = df[col].apply(convert_str_to_list)

    # Calculer la taille des produits affichés et des produits cliqués.
    df["product_skus_size"] = df["product_skus_hash"].apply(lambda x: len(x) if isinstance(x, list) else 0)
    df["clicked_skus_size"] = df["clicked_skus_hash"].apply(lambda x: len(x) if isinstance(x, list) else 0)

    # Ajouter les produits cliqués dans les produits affichés, seulement s'ils sont absents.
    def add_clicked(row):
        if isinstance(row["clicked_skus_hash"], list):
            return list(set(row["product_skus_hash"] or []) | set(row["clicked_skus_hash"]))
        return row["product_skus_hash"]

    df["updated_product_skus_hash"] = df.apply(add_clicked, axis=1)

def replace_low_interaction_products(df, column_name='product_url_hash', threshold=5, common_id='common_product'):
    # Comptage des interactions pour chaque produit

Normalisation des vecteurs​

Nous avons ont utilisé les vecteurs pré-entraînés fournis dans le jeu de données, qui comprennent des vecteurs de requêtes de recherche (query_vector), des descriptions de produits (description_vector), et des images de produits (image_vector). Nous avons appliqué une normalisation L2 aux vecteurs plutôt que d'utiliser les vecteurs originaux avec une normalisation par couche. La normalisation L2 permet de maintenir l'échelle des caractéristiques similaire tout en préservant les relations de similarité entre les produits similaires​​.

La normalisation L2, également connue sous le nom de normalisation euclidienne, consiste à ajuster les valeurs d'un vecteur de sorte que la longueur (ou la norme) du vecteur soit égale à 1. Elle se fait selon la formule suivante :

vecteur_normaliseˊ=vecteur∥vecteur∥2\text{vecteur}\_\text{normalisé} = \frac{\text{vecteur}}{\| \text{vecteur} \|_2}vecteur_normaliseˊ=∥vecteur∥2​vecteur​

où ∥vecteur∥2\| \text{vecteur} \|_2∥vecteur∥2​ est la norme L2 du vecteur, calculée comme la racine carrée de la somme des carrés de ses éléments. En d'autres termes, pour un vecteur v=[v1,v2,…,vn]\mathbf{v} = [v_1, v_2, \ldots, v_n]v=[v1​,v2​,…,vn​], la norme L2 est calculée par :

∥v∥2=v12+v22+…+vn2\| \mathbf{v} \|_2 = \sqrt{v_1^2 + v_2^2 + \ldots + v_n^2}∥v∥2​=v12​+v22​+…+vn2​​

Une fois la norme L2 calculée, chaque élément du vecteur est divisé par cette valeur pour obtenir le vecteur normalisé.

Concrètement, le code calcule d'abord la norme L2 du vecteur image_vector à l'aide de np.linalg.norm avec l'ordre 2 (norme euclidienne), puis divise chaque élément du vecteur par cette norme pour obtenir le vecteur normalisé.

def normalize_vector(vect, max_features: int):
    # Normaliser un vecteur en utilisant la normalisation L2 et limiter sa longueur
    # à 'max_features'.

    # Gérer les cas où vect est None, un entier, ou autre chose que np.ndarray
    if vect is None or not isinstance(vect, np.ndarray):
        return np.zeros(max_features)

    # S'assurer que vect est un tableau NumPy 1D
    if vect.ndim != 1:
        raise ValueError("Le vecteur doit être un tableau NumPy 1D.")

    # Sélectionner les premiers 'max_features' éléments
    if len(vect) > max_features:
        vect = vect[:max_features]
    elif len(vect) < max_features:
        # Compléter les vecteurs plus courts avec des zéros
        vect = np.pad(vect, (0, max_features - len(vect)), "constant")

    # Normalisation L2
    norm = np.linalg.norm(vect, 2)
    normalized_vect = vect / norm if norm != 0 else vect

    return normalized_vect

def prepare_data(sessions: list, non_text_features_all: list, max_seq_length: int):
    text_sequences = []
    non_text_features_prepared = []
    labels = []
    futures = []

    for session, non_text_features in zip(sessions, non_text_features_all):
        # Boucle pour chaque fenêtre de 4 interactions dans la session
        # Requis pour éviter des dépassements de capacité mémoire.
        for i in range(len(session) - 4):
            # Création de la séquence de texte
            # Insérer un "|" entre chaque interaction et un ";" entre chaque caractéristique.
            text_seq = "|".join([";".join([str(field) for field in interaction[:-1]]) for interaction in session[i:i+4]])

            if max_seq_length > 0:
                if len(text_seq) > max_seq_length:
                    text_seq = text_seq[:max_seq_length]  # Tronquer
                else:
                    text_seq += " " * (max_seq_length - len(text_seq))  # Remplir avec des espaces

            text_sequences.append(text_seq)

            # Traitement des caractéristiques non textuelles pour l'interaction suivante
            non_text_vector = non_text_features[i+4]
            non_text_features_prepared.append(non_text_vector)

            # L'étiquette est la première colonne de l'interaction suivante après la fenêtre
            label = int(session[i+4, 0])
            labels.append(label)

            # Les interactions futures sont dans la première colonne des interactions suivantes.
            # Limiter le nombre d'interactions futures à 20.
            future_interactions = session[i+4:i+24, 0]
            futures.append(future_interactions)

    return text_sequences, non_text_features_prepared, labels, futures

Tokenisation des séquences de texte​

Les paramètres utilisés dans la méthode encode_plus de la classe XLNetTokenizer de la bibliothèque transformers sont :

  1. seq: C'est la séquence de texte que que nous voulons tokeniser. Dans notre cas, seq représente chaque élément de text_sequences.

  2. add_special_tokens: Lorsque ce paramètre est défini sur True, le tokenizer ajoutera des tokens spéciaux au début et à la fin de chaque séquence. Pour XLNet, cela inclut généralement les tokens de début et de fin de séquence.

  3. max_length: Ceci définit la longueur maximale d'une séquence tokenisée. Si une séquence est plus longue que cette valeur, elle sera tronquée à max_length.

  4. truncation: Lorsqu'il est défini sur True, cela permet au tokenizer de tronquer les séquences qui dépassent max_length. C'est nécessaire pour normaliser la longueur des séquences, ce qui est important pour les modèles basés sur les transformers qui nécessitent une longueur de séquence fixe.

  5. padding: Ce paramètre contrôle comment le padding est ajouté aux séquences plus courtes que max_length. Quand il est réglé sur "max_length", le tokenizer ajoutera du padding à la séquence jusqu'à ce qu'elle atteigne max_length. Le padding est généralement ajouté à la fin de la séquence.

  6. return_attention_mask: Si réglé sur True, le tokenizer retournera un masque d'attention en plus des IDs de tokens. Le masque d'attention a des 1 pour chaque token significatif et des 0 pour tout padding ajouté, indiquant au modèle les parties de la séquence à ignorer pendant l'entraînement ou l'inférence.

Les valeurs retournées par encode_plus dans votre code sont :

  • input_ids: Une liste d'identifiants de tokens représentant la séquence de texte après tokenisation.
  • attention_masks: Le masque d'attention correspondant, indiquant au modèle quels tokens doivent être pris en compte et quels tokens sont des paddings.
from transformers import XLNetTokenizer

def tokenize_sequences(text_sequences, tokenizer, max_seq_length: int):
    input_ids = []
    attention_masks = []

    for seq in text_sequences:
        encoded_seq = tokenizer.encode_plus(
            seq,
            add_special_tokens=True,  # Ajouter des tokens spéciaux
            max_length=max_seq_length,  # Longueur maximale de la séquence
            truncation=True,  # Tronquer si la séquence est plus longue que max_length
            padding="max_length",  # Ajouter du padding pour atteindre max_length
            return_attention_mask=True,  # Renvoyer le masque d'attention
        )
        input_ids.append(encoded_seq["input_ids"])
        attention_masks.append(encoded_seq["attention_mask"])

    return input_ids, attention_masks

        def __init__(self, model_name, num_classes, combined_vectors_length, dropout_rate: float):
            super(XLNetForSequencePrediction, self).__init__()
            self.xlnet = XLNetModel.from_pretrained(model_name)
            self.non_text_feature_layer = nn.Linear(combined_vectors_length, self.xlnet.config.hidden_size)

            # Ajouter une couche de Dropout
            self.dropout = nn.Dropout(dropout_rate)

            self.output_layer = nn.Linear(self.xlnet.config.hidden_size, num_classes)

        def forward(self, input_ids, attention_mask, non_text_features):
            # Traitement des entrées textuelles avec XLNet
            xlnet_output = self.xlnet(input_ids=input_ids, attention_mask=attention_mask)
            text_embedding = xlnet_output.last_hidden_state[:, 0, :]  # Utiliser le CLS token

            non_text_embedding = self.non_text_feature_layer(non_text_features)

            # Combinaison des embeddings textuels et non textuels
            combined_embedding = text_embedding + non_text_embedding

            # Application du Dropout
            combined_embedding = self.dropout(combined_embedding)

            # Passage à travers la couche de sortie
            output = self.output_layer(combined_embedding)

            return output

        def _init_weights(self, module):
            """Initialise les poids du module."""
            if isinstance(module, nn.Linear):

Mean Reciprocal Rank (MRR@20)​

La métrique Mean Reciprocal Rank (MRR), mentionnée dans l'article de Coveo, est fréquemment utilisée dans les systèmes de recommandation pour évaluer la performance d'un modèle, particulièrement dans des contextes où l'ordre des recommandations est crucial.

Le MRR est une mesure qui considère le rang de la première réponse pertinente dans une liste de réponses prédites. Il se définit comme la moyenne des inverses des rangs de ces premières réponses pertinentes. Voici sa méthode de calcul :

  1. Réciproque du rang : Pour chaque question ou requête (dans notre contexte, chaque séquence d'interactions utilisateur), il faut identifier le rang de la première prédiction pertinente. Ce rang correspond à la position de la prédiction pertinente dans la liste triée de recommandations, en commençant par 1. On calcule ensuite le réciproque de ce rang. Par exemple, si la première réponse pertinente est en deuxième position, le réciproque du rang est 1/2 = 0.5.

  2. Moyenne des réciproques des rangs : La moyenne de ces valeurs réciproques est ensuite calculée sur l'ensemble des questions ou requêtes.

Pour calculer le MRR, nous avons donc besoin d'une liste de prédictions du modèle, où ces prédictions sont classées par ordre de probabilité décroissante. Pour un ensemble de NNN interactions futures, où rir_iri​ est le rang de la première interaction réelle future dans la liste des interactions prédites classées par ordre de probabilité décroissante, le MRR se calcule de la manière suivante :

MRR=1N∑i=1N1ri\text{MRR} = \frac{1}{N} \sum_{i=1}^{N} \frac{1}{r_i}MRR=N1​i=1∑N​ri​1​

Le MRR est une mesure qui favorise les modèles capables de placer les interactions pertinentes le plus haut possible dans la liste des prédictions. Un MRR élevé indique que le modèle est performant dans l'identification de ce qui est pertinent pour l'utilisateur dès les premières prédictions.

Precision@20, Recall@20 et F1 score​

L'article de Coveo mentionne également les trois métriques suivantes.

La precision@20 est le nombre de recommandations correctes dans les top-20 recommandations pour une session donnée, divisé par 20. La précision moyenne à 20 sur toutes les sessions est la somme de la précision pour chaque session divisée par le nombre total de sessions. Voici la formule correcte :

Precision@20=1∣S∣∑s=1∣S∣∣reˊel(s)∩predk(s)∣k\text{Precision@20} = \frac{1}{|S|} \sum_{s=1}^{|S|} \frac{| \text{réel}(s) \cap \text{pred}_k(s) |}{k}Precision@20=∣S∣1​s=1∑∣S∣​k∣reˊel(s)∩predk​(s)∣​

où ( |S| ) est le nombre total de sessions et ( k ) est le nombre de recommandations à considérer, ici 20.

Le recall@20 est le nombre de recommandations correctes dans les top-20 recommandations pour une session donnée, divisé par le nombre total d'éléments pertinents dans cette session. Le rappel moyen à 20 sur toutes les sessions est la somme du rappel pour chaque session divisée par le nombre total de sessions. Voici la formule correcte :

Recall@20=1∣S∣∑s=1∣S∣∣reˊel(s)∩predk(s)∣∣reˊel(s)∣\text{Recall@20} = \frac{1}{|S|} \sum_{s=1}^{|S|} \frac{| \text{réel}(s) \cap \text{pred}_k(s) |}{|\text{réel}(s)|}Recall@20=∣S∣1​s=1∑∣S∣​∣reˊel(s)∣∣reˊel(s)∩predk​(s)∣​

où ∣reˊel(s)∣|\text{réel}(s)|∣reˊel(s)∣ est le nombre d'éléments pertinents dans la session sss.

Le score F1 est une mesure qui combine la précision et le rappel. Il est particulièrement utile lorsqu'il est nécessaire de trouver un équilibre entre la précision et le rappel. Le score F1 est défini comme la moyenne harmonique de la précision et du rappel :

F1 Score=2×Precision×RecallPrecision+Recall\text{F1 Score} = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}F1 Score=2×Precision+RecallPrecision×Recall​

où la Precision et le Recall sont les valeurs calculées pour une session individuelle, et le score F1 est la moyenne de ces scores sur toutes les sessions.

def convert_list_to_top_K(items_list: list, topK: int):
    """
    Extract top_K items
    It is assumed that the list of items are sorted in descending order of importance/weight/relevance

    :param items_list: list where element is a list of items
    :param topK: top K limit
    :return: list of items shortened with top K
    """
    return [items[:topK] for items in items_list]

def mrr_at_k(preds: list, labels: list, topK: int):
    assert len(labels) > 0
    assert len(preds) == len(labels)

    # get top K predictions
    converted_preds = convert_list_to_top_K(items_list=preds, topK=topK)
    rr = []
    for p, l in zip(converted_preds, labels):
        if len(l) == 0:
            rr.append(0.0)
        else:
            # get next_item from labels
            next_item = l[0]
            # add 0.0 explicitly if not there (for transparency)
            if next_item not in p:
                rr.append(0.0)
            # else, take the reciprocal of prediction rank
            else:
                rr.append(1.0 / (p.index(next_item) + 1))

    # return the mean reciprocal rank
    return sum(rr) / len(labels)

def f1_at_k(preds: list, labels: list, topK: int):
    assert len(labels) > 0
    assert len(preds) == len(labels)

    # get top K predictions
    converted_preds = convert_list_to_top_K(items_list=preds, topK=topK)
    all_precision = []
    all_recall = []
    all_f1 = []

    # for each recommendation prediction, calculate the f1 based on ground truth
    for p, l in zip(converted_preds, labels):
        nb_hits = len(set(p).intersection(set(l)))
        precision = nb_hits / topK
        recall = nb_hits / len(set(l)) if len(l) > 0 else 0.0
        f1 = (2 * precision * recall) / (precision + recall) if (precision + recall) > 0 else 0.0
        all_precision.append(precision)
        all_recall.append(recall)
        all_f1.append(f1)

    # get average f1 across all predictions
    avg_f1 = sum(all_f1) / len(labels)
    return avg_f1

def next_item_metric(preds: list, labels: list, top_K: int = 20):

    :param top_K: as per README, by default we consider metric @ K=20
    :return: mrr for next item prediction

    :param labels:
    :param top_K: as per README, by default we consider metric @ K=20

            optimizer.step()
